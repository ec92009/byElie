# byElie Review — 2026-05-01

Scope: the static portfolio site shipped from this repo.
- **Main "program":** the six top-level HTML pages (`index.html`, `plans.html`, `prints.html`, `gallery.html`, `cad-design.html`, `web-design.html`) + `shared.css` + `styles.css`. There's no build step, no JS bundler, no backend — the pages *are* the program.
- Treating "Looks / Behavior / Architecture" through that lens.

Reviewed at v54.19.

---

## The Site

### Looks (UI)
**Strengths**
- Confident, restrained design language. Monochrome palette driven by CSS custom properties (`--bg`, `--surface`, `--text`, `--muted`, `--line`, `--brand`) in `shared.css:1–29`, with a clean light/dark token swap at `data-theme="light"`. This is the right primitive — palette, radii, gutters, panel rhythm all derive from variables, not hard-coded values.
- Typography is single-family (`Space Grotesk`) with fluid sizing (`clamp(2rem, 4vw, 3.2rem)` on `h1`, `shared.css:65`). Reads consistently from phone to ultrawide without a media-query mess.
- Sticky topbar with `backdrop-filter: blur(10px)` and a token-mixed translucent background (`shared.css:48–51`) is a small but expensive-feeling touch.
- The hand-drawn SVG cocker spaniel (`index.html:71+`) as a carousel companion is genuinely charming and ties the brand to the byElie identity. Worth keeping.
- Pill-shaped nav, theme toggle, and CTAs (`border-radius:999px` throughout) give the site a coherent "rounded-pill + sharp-panel" rhythm — `--radius:0` on panels intentionally contrasts with the pills.

**Weaknesses**
- **Six near-identical HTML pages.** Each sub-page is 506 lines and only differs from `plans.html` in `<title>`, the active nav link, and the page body. The boilerplate (head, fonts, header, theme-toggle, footer) is duplicated six times. One template + a build step (or even just SSI / a tiny generator) would cut the codebase by ~70%.
- **Inline `<style>` block in `index.html`** (`index.html:24–40`) for hero animations and `.brand`/`.site-logo`. Either move into `styles.css` or a dedicated `home.css` — the rule "shared.css for shared, styles.css for overrides" is already established; the inline block breaks it.
- **Version string baked into HTML** (`BY ELIE — v54.19` at `index.html:50`). The README's `?v=54.18` cache-bust on `shared.css`/`styles.css` and the `v54.19` brand display already drift by one bump. Either centralize or drop the per-page badge.
- **Cache-bust query (`?v=54.18`)** is hand-edited per-page. Six pages × two stylesheets = twelve sites of truth for one number.
- The SVG spaniel is inlined ~80 lines into every page that uses it. Extract to `assets/spaniel.svg` and reference, or accept that the spaniel only lives on `index.html` (which seems to be the case — worth verifying).
- Light theme has the brand color as `#111111` (near-black on `#fafafa`) — fine. Dark theme uses `--brand:#f0f0f0` on `#0e0e0e` — also fine. But `--brand-2:#909090` at 4.5:1 contrast on the dark background is borderline for body text on small screens. Worth a quick a11y check.

### Behavior (UX)
**Strengths**
- Theme persistence via `localStorage` set in a tiny inline `<script>` *before* `shared.css` loads (`index.html:8–15`). This is the right pattern — it prevents a flash-of-wrong-theme on reload. Good instinct.
- `prefers-reduced-motion: reduce` honored for hero animations (`index.html:37–39`). Most portfolios ignore this.
- Skip-to-content link (`.skip-link`) at the top of `body`. Small a11y win that most one-off sites skip.
- No analytics, no cookie banner, no popups. The site loads, works, and gets out of the way. For a creator portfolio this is the correct default.
- "Request quote" anchor (`#contact`) on the hero CTA — the contact path is one click from anywhere.

**Weaknesses**
- **Carousel pagination/keyboard behavior** isn't visible in the HTML I read — `data-offers-wrap tabindex="0"` and `data-offers-track` suggest JS hooks, but I didn't see the JS. If the carousel relies on external/inline JS lower in the file, fine; if not, the keyboard story may be broken.
- **No `lang` attribute on most pages** — wait, `index.html` has `lang="en"`. Verify the other five do too; copy-paste drift is the usual culprit.
- `data-href` on `.carousel-item` (`index.html:65–69`) suggests JS turns the cards into links. A plain `<a class="card">…</a>` would work without JS, be keyboard-accessible by default, and be readable by screen readers as links. Worth refactoring.
- The version pill in the header (`BY ELIE — v54.19`) is a developer affordance, not a user one. Consider hiding it from the public site or moving it to a `<meta>` tag.
- "Night/Day" theme toggle labels (`shared.css:60–62`) are clever but ambiguous on first encounter — does "Night" mean "switch to night mode" or "currently night"? The convention varies. A sun/moon icon plus a tooltip would be unambiguous.

### Architecture
**Strengths**
- **Zero build step.** Open the folder in a browser, it works. `python3 -m http.server` for local preview. This is the right choice for a portfolio — every pound of toolchain you'd add is a pound the site has to carry.
- **Token-driven CSS** in `shared.css` with a clean light/dark switch. Adding a new theme would be a 15-line `[data-theme="X"]` block.
- **Separation of concerns** between `shared.css` (390 lines, layout + components + theme) and `styles.css` (35 lines, page-specific overrides). Good ratio — the small `styles.css` is a sign the shared layer is doing its job.
- Assets folder is tight (4 files), no orphan images.

**Weaknesses**
- **Resolved:** GitHub Pages is configured to serve from `/` on `main`, so the orphan `docs/` duplicate was removed. The root files are the deploy source.
- **No template engine.** With six pages × ~500 lines, a 50-line template + 6 small content files would be a clear win. Options that don't break "no build step": Eleventy, Astro (heavier), or even a 30-line Python script that stitches a `<head>` partial + nav + content into the six output files. Worth the half-day investment.
- **Cache-bust version is hand-bumped per-page.** A trivial pre-commit hook could rewrite `?v=X.Y` everywhere from a single source.
- **No sitemap.xml, no robots.txt, no Open Graph meta tags** that I saw. For a service portfolio that wants to rank for "CAD on demand" or "3D print plans," that's leaving table-stakes SEO on the floor.
- **Nothing tracks deploys/changes.** No `CHANGELOG.md`, no version reflected in a meta tag. If you want to know "did the live site update?", you have to read the brand pill.

---

## Cross-cutting recommendations (priority order)

1. **Eliminate the six-page duplication.** Either build-step it (Eleventy is 10 minutes to set up) or write a 40-line `build.py` that emits the six files from a template + content folder. Biggest leverage available.
2. **Resolved: remove orphan `docs/` duplication.** GitHub Pages serves from `/` on `main`, so there is no mirror to maintain.
3. **Centralize the version + cache-bust string.** One source of truth, propagated by the build (or by the same `build.py`).
4. **Add SEO basics.** `<meta name="description">`, Open Graph tags, `sitemap.xml`, `robots.txt`. For a portfolio that asks for quote requests, this matters.
5. **Replace `data-href` cards with real `<a>` tags.** Zero-JS keyboard accessibility for the carousel.
6. **Audit `lang` attribute and skip-link presence on all six pages** — copy-paste drift is the likely failure mode.

## What I would NOT change

- The zero-build, vanilla-HTML+CSS posture. Don't migrate to React/Next/Astro just because. The site is fast, transparent, and forkable, and that *is* the brand.
- The CSS custom-property design tokens. Already the right primitive.
- The hand-drawn SVG spaniel. Charming, distinctive, on-brand.
- Single-font choice. Restraint is correct here.
- The light/dark theme + persistence pattern. Idiomatic, fast, no flash.
- No analytics / no cookie banner. For a portfolio site this is the dignified default.
